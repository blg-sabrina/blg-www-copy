BigLedger 中的价格集提供精细化的、基于规则的定价能力，能够实现适应各种业务状况、客户细分和运营场景的复杂定价策略。这一高级定价系统凭借强大的逻辑引擎和灵活的配置选项，支持企业级需求。

概述

价格集是 BigLedger 定价架构中最高级的层次，提供能够根据可配置条件自动应用不同定价策略的基于规则的逻辑。它们与价格簿和定价方案无缝协作，创建全面、智能的定价解决方案。

ℹ️
基础架构：价格集在 价格簿 → 定价方案 → 价格集 → 项目价格 的层级结构中运作，提供决定在特定业务条件下应用何种价格的逻辑层。

了解价格集架构

核心组件与关系

层级结构：

  • 价格簿：定价信息的组织容器
  • 定价方案：定义价格计算方法的模板
  • 价格集：用于条件定价的基于规则的逻辑引擎
  • 定价规则：控制价格应用的具体条件和处理方式

业务逻辑流程：

  1. 条件评估：系统评估单据表头和行项目条件
  2. 规则匹配：根据业务规则识别适用的价格集
  3. 优先级应用：当多个价格集匹配时应用最高优先级规则
  4. 处理执行：根据配置的处理方式计算最终价格
  5. 验证：确保定价完整性和业务规则合规性

价格集与其他定价组件的比较

价格集与定价方案：

  • 定价方案：定义价格类型和计算方法的静态模板
  • 价格集：根据业务逻辑有条件地应用定价的动态规则引擎

价格集与价格簿：

  • 价格簿：将相关定价信息分组的组织容器
  • 价格集：决定何时以及如何应用特定定价的智能逻辑系统

高级能力：

  • 条件逻辑：复杂的 AND/OR 规则组合
  • 优先级管理：具有覆盖能力的层级规则应用
  • 性能优化：面向大规模运营的高效规则评估
  • 集成支持：与所有 BigLedger 模块的无缝运作

价格集创建与配置

访问价格集管理

导航与设置

  1. 从定价管理菜单导航到 价格集配置（Price Set Configuration）
  2. 确保前置组件已配置：
    • 价格簿：已创建并正确配置
    • 定价方案：已定义适当的价格类型
    • 项目主数据：具有基础定价信息的产品
  3. 访问价格集创建界面

前置条件与依赖项

必需组件：

  • 启用的价格簿：价格集必须与现有价格簿关联
  • 定价方案：必须配置至少一个定价方案
  • 用户权限：价格集管理的适当访问权限
  • 系统资源：足够的系统容量用于规则处理

价格集创建流程

核心配置信息

价格集标识：

  • 价格簿选择：选择将包含此价格集的价格簿
  • 价格集代码：系统生成的唯一标识符（创建后只读）
  • 价格集名称：便于识别和管理的描述性名称
  • 描述：对价格集用途和业务逻辑的详细说明

优先级与状态管理：

  • 优先级级别：规则应用顺序的数字优先级（1 = 最高优先级）
  • 状态：用于受控部署的启用、停用或草稿状态
  • 生效日期：用于时效性定价的可选日期范围
  • 覆盖权限：定义谁可以覆盖这些定价规则

高级配置选项

性能设置：

  • 规则复杂度：在精细逻辑和系统性能之间取得平衡
  • 缓存选项：为频繁访问的价格集配置缓存
  • 验证级别：设置规则冲突的验证严格程度
  • 监控：为复杂规则集启用性能监控

集成设置：

  • 模块集成：配置与 POS、电子商务和其他模块的集成
  • API 访问：启用外部系统访问价格集逻辑
  • 实时处理：配置实时与批量定价计算
  • 审计日志：为价格集规则应用启用详细日志记录

规则配置与逻辑

单据表头规则

单据表头规则在整体交易层面评估条件，考虑客户信息、交易日期和业务背景。

基于客户的规则

客户分类规则：

  • 客户类型：根据零售、批发或 VIP 分类应用不同定价
  • 客户组：针对特定客户组或市场细分
  • 客户位置：基于地理位置的区域市场差异化定价
  • 客户关系：基于客户资历、销量或关系状态的定价

客户规则配置示例：

Rule Name: VIP Customer Discount
Condition: Customer.Type = "VIP" AND Customer.TotalPurchases > $10,000
Treatment: Apply 15% discount to all items
Priority: 1

高级客户逻辑：

  • 购买历史：基于客户购买模式的规则
  • 信用状态：基于付款条款和信用评级的定价调整
  • 忠诚度计划：与客户忠诚度和奖励计划的集成
  • 合同定价：为签订合同的客户提供的特殊定价

基于日期和时间的规则

时间定价逻辑：

  • 季节性定价：旺季和淡季的不同定价
  • 促销期：限时特价活动
  • 星期几：工作日与周末的不同定价
  • 一天中的时段：某些业务模式的按小时定价

日期规则示例：

Rule Name: Holiday Season Premium
Condition: Date BETWEEN "2024-11-15" AND "2024-12-31"
Treatment: Apply 10% markup to selected categories
Priority: 2

Rule Name: Off-Season Discount
Condition: Date BETWEEN "2024-01-15" AND "2024-03-15"
Treatment: Apply 20% discount to seasonal items
Priority: 3

实体和业务背景规则

组织规则：

  • 销售渠道：零售、在线、批发渠道的不同定价
  • 业务单元：部门或部门特定的定价规则
  • 位置：基于门店或仓库位置的定价
  • 货币：具有自动转换的多币种定价

业务背景示例：

  • 订单规模：批发定价的最低订单数量
  • 付款条款：现金与信用定价差异
  • 配送方式：自提与配送的定价调整
  • 加急订单：加急处理的高端定价

行项目规则

行项目规则在单个产品或服务层面评估条件，实现精细化的产品特定定价逻辑。

产品特定规则

项目分类规则：

  • 产品类别：基于类别的定价规则和调整
  • 品牌：品牌特定的定价策略和毛利
  • 产品生命周期：新产品、成熟产品和生命末期产品的不同定价
  • 库存状态：基于库存水平和周转率的定价调整

产品规则配置：

Rule Name: Electronics Category Discount
Condition: Item.Category = "Electronics" AND Quantity >= 5
Treatment: Apply 8% discount
Priority: 1

Rule Name: Clearance Pricing
Condition: Item.Status = "Discontinued" AND Inventory.Level < 10
Treatment: Apply 30% discount
Priority: 2

基于数量的规则

批量定价逻辑：

  • 数量档次：基于订单数量的分层定价
  • 批量折扣：为较大订单提供的递进折扣
  • 混合数量：不同产品组合的规则
  • 最低订单：最低订单数量的定价要求

数量档次示例：

Rule Name: Volume Discount Tier 1
Condition: Quantity BETWEEN 10 AND 49
Treatment: Apply 5% discount
Priority: 1

Rule Name: Volume Discount Tier 2
Condition: Quantity BETWEEN 50 AND 99
Treatment: Apply 10% discount
Priority: 1

Rule Name: Volume Discount Tier 3
Condition: Quantity >= 100
Treatment: Apply 15% discount
Priority: 1

高级项目逻辑

复杂产品规则：

  • 跨类别：影响多个产品类别的规则
  • 互补产品：相关或捆绑商品的定价
  • 替代逻辑：主要商品不可用时的替代产品定价
  • 基于属性：基于特定产品属性或规格的规则

正则表达式匹配：

  • 模式匹配：使用正则表达式匹配复杂的项目代码模式
  • 动态分类：基于项目属性的自动分类
  • 灵活分组：基于命名约定或代码对项目分组
  • 异常处理：针对不符合标准模式的项目的特定规则

处理配置与计算

处理类型和方法

价格集处理定义了在规则条件满足后如何计算和应用定价调整。

折扣处理

百分比折扣：

  • 固定百分比：应用一致的百分比折扣（例如，八折优惠）
  • 递进百分比：基于订单价值的递进折扣
  • 类别特定：不同产品类别的不同折扣百分比
  • 客户特定：基于客户分类的个性化折扣百分比

金额折扣：

  • 固定金额：特定的货币折扣（例如，减 $50）
  • 按单位折扣：每购买一件商品的固定折扣
  • 基于阈值：在特定购买水平激活的金额折扣
  • 递进金额：为较大订单提供的递增金额折扣

加价处理

成本加成定价：

  • 固定加价：在成本基础上的一致加价百分比
  • 可变加价：基于产品类别或客户类型的不同加价
  • 竞争性加价：基于市场的加价计算
  • 基于价值的加价：基于客户感知价值的加价

市场定价：

  • 高端定位：为高端产品或服务设置较高加价
  • 渗透定价：为市场进入或增长设置较低加价
  • 竞争平价：旨在匹配竞争对手定价的加价
  • 动态调整：基于市场状况的实时加价调整

固定价格处理

绝对定价：

  • 合同定价：与特定客户协商的固定价格
  • 促销定价：营销活动的特殊固定价格
  • 清仓定价：用于库存清仓的固定低价
  • 捆绑定价：产品或服务捆绑的固定价格

条件固定定价：

  • 基于数量的固定：在特定数量激活的固定价格
  • 限时固定：特定时间段的固定价格
  • 客户特定固定：为关键账户协商的固定价格
  • 渠道特定固定：不同销售渠道的不同固定价格

计算方法与公式

标准计算公式

百分比计算：

Final Price = Base Price × (1 ± Percentage/100)
Example: $100 × (1 - 15/100) = $85 (15% discount)

金额计算：

Final Price = Base Price ± Fixed Amount
Example: $100 - $25 = $75 (fixed $25 discount)

复合计算：

Multiple Rules: Apply in priority order
Rule 1: 10% discount = $100 × 0.9 = $90
Rule 2: $5 discount = $90 - $5 = $85

高级计算逻辑

条件计算：

  • 分层计算：不同数量或价值档次的不同计算方法
  • 递进计算：基于累计订单价值变化的计算
  • 复利：金融产品或服务的复杂计算
  • 矩阵定价：基于多个变量的多维定价

自定义公式支持：

  • 业务规则集成：包含特定业务逻辑的自定义公式
  • 外部数据集成：使用外部市场数据或费率的计算
  • 实时调整：根据当前市场状况调整的公式
  • 验证逻辑：内置验证以确保计算准确性和业务规则合规性

复杂业务场景

多层级分销定价

企业分销场景通常需要精细的定价结构，以适应多层级分销渠道、区域市场和客户关系。

分销渠道层级

渠道特定定价策略：

  • 制造商到经销商：带批量激励的批发定价
  • 经销商到零售商：带营销支持的零售批发定价
  • 零售商到消费者：带市场定位的终端用户定价
  • 直接销售：带渠道冲突管理的制造商直接定价

地理市场差异化：

  • 区域定价：不同地理市场的不同定价
  • 货币本地化：带自动转换的本地货币定价
  • 税务合规：不同司法管辖区的自动税费计算
  • 物流调整：运输和处理成本的定价调整

复杂分销规则

示例：多层级电子产品分销：

Rule Set 1: Distributor Pricing
Condition: Customer.Type = "Distributor" AND Order.Value >= $50,000
Treatment: Apply distributor pricing schedule (40% off MSRP)
Priority: 1

Rule Set 2: Volume Incentives
Condition: Customer.Type = "Distributor" AND Quarterly.Volume >= $200,000
Treatment: Additional 5% discount
Priority: 2

Rule Set 3: Regional Adjustment
Condition: Customer.Region = "Asia-Pacific" AND Product.Category = "Electronics"
Treatment: Apply 8% regional adjustment
Priority: 3

季节性与促销定价

季节性定价策略需要精细的基于时间的规则，能够根据日历事件、库存水平和市场状况自动调整定价。

季节性定价策略

节日和活动定价：

  • 季前铺垫：逐步提价以迎接旺季
  • 旺季高端：高需求期间的最高定价
  • 季后清仓：积极折扣以清理季节性库存
  • 淡季维持：保持市场存在的基准定价

库存驱动的季节性定价：

  • 库存水平集成：基于当前库存水平的定价调整
  • 周转率：基于库存周转速度的定价修改
  • 预测集成：基于需求预测的预测性定价
  • 自动调整：基于库存指标的实时价格变化

动态促销定价

基于活动的定价：

Rule Set: Black Friday Electronics Sale
Condition: Date = "2024-11-29" AND Category = "Electronics"
Treatment: Apply 25% discount with maximum $500 discount per item
Priority: 1
Effective: 2024-11-29 00:00 to 2024-11-29 23:59

Condition: Product.Code LIKE "PROMO_*" AND Quantity >= 2
Treatment: Apply 50% discount to lowest-priced item
Priority: 2
Effective: 2024-12-01 to 2024-12-31

B2B 门户与电子商务集成

现代企业需要能够在多个数字渠道之间无缝运作的定价系统，同时保持一致性并支持渠道特定策略。

B2B 门户定价

基于账户的定价：

  • 客户特定目录：带客户特定定价的个性化产品目录
  • 合同定价集成：自动应用协商的合同条款

自助定价工具：

  • 报价生成：客户能够生成带实时定价的报价单
  • 价格历史：访问历史定价以进行预算规划和分析

电子商务定价集成

多平台同步：

  • 实时更新：所有电子商务平台的即时定价更新
  • 平台特定规则：不同在线平台的不同定价策略
  • 竞争情报：与竞争定价监控工具的集成
  • 动态调整：基于市场状况的自动定价调整

客户体验增强：

  • 个性化定价：登录时显示客户特定定价
  • 基于数量的显示：实时显示数量档次定价

  • 移动优化：跨所有设备类型的一致定价体验

优先级管理与规则冲突解决

了解优先级系统

优先级管理确保当多个价格集可能应用于同一交易时，系统以可预测的、符合业务逻辑的方式应用最合适的定价。

优先级层级

数字优先级系统：

  • 优先级 1：最高优先级规则（最具体、最关键的业务规则）
  • 优先级 2-5：标准业务规则的中级优先级
  • 优先级 6-10：一般规则和默认设置的较低优先级
  • 默认处理：当没有特定规则匹配时的系统行为

优先级分配策略：

  • 客户特定：协商客户定价的最高优先级
  • 促销：时效性促销活动的高优先级
  • 基于数量：基于数量的定价规则的中等优先级
  • 一般：类别或一般市场定价的较低优先级

冲突解决逻辑

规则重叠管理：

Example Conflict Scenario:
Rule A (Priority 1): VIP Customer gets 20% discount
Rule B (Priority 2): Electronics category gets 15% discount
Rule C (Priority 3): Orders over $1000 get 10% discount

Resolution: VIP customer buying electronics over $1000 gets 20% discount (Priority 1 wins)

组合与覆盖：

  • 覆盖逻辑：较高优先级规则完全覆盖较低优先级规则
  • 累加逻辑：某些规则可以组合以产生累积效果
  • 排他逻辑：某些规则不能组合（冲突预防）
  • 业务逻辑验证：系统验证以防止无意义的组合

复杂业务规则

多维优先级：

  • 客户 + 产品：组合的客户和产品特定规则
  • 时间 + 数量：时间规则与基于数量的规则组合
  • 地理 + 渠道：位置和销售渠道组合规则
  • 合同 + 促销：带促销叠加的协商定价

  • 系统维护：系统维护或更新期间的定价行为
  • 数据验证：处理不完整或无效数据场景

故障排查与常见问题

规则配置问题

常见规则问题

逻辑错误：

  • 问题：由于逻辑错误，规则未按预期评估
  • 常见原因：错误的 AND/OR 逻辑、缺少括号、错误的数据类型
  • 解决方案：
    • 使用规则测试工具验证逻辑
    • 将复杂规则拆分为更简单的组件
    • 清晰记录规则逻辑以供审查
    • 实施逐步规则评估日志记录
  • 预防：建立规则开发标准和同行评审流程

优先级冲突：

  • 问题：由于优先级冲突导致意外定价结果
  • 常见原因：重叠的优先级级别、不明确的优先级分配
  • 解决方案：
    • 审查和重构优先级分配
    • 实施优先级测试场景
    • 创建优先级文档和指南
    • 为复杂场景使用优先级模拟工具
  • 预防：制定清晰的优先级分配方法

数据和集成问题

数据同步问题：

  • 问题：定价规则引用过时或不正确的数据
  • 常见原因：延迟的数据更新、集成故障、缓存问题
  • 解决方案：
    • 实施实时数据验证
    • 创建数据同步监控
    • 建立数据刷新程序
    • 提供手动数据覆盖能力
  • 预防：定期数据完整性检查和自动监控

性能下降：

  • 问题：由于复杂规则集导致定价计算缓慢
  • 常见原因：低效规则、大型数据集、硬件不足
  • 解决方案：
    • 优化规则逻辑和条件
    • 实施智能缓存策略
    • 审查和升级系统资源
    • 在适当情况下实施并行处理
  • 预防：在规则开发期间进行性能测试
